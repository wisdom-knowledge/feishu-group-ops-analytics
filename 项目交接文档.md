## 结论

这个项目当前是“能用、数据已跑通、看板和问数助手可演示”的内部运营分析工具，但还不是一个工程化很干净的生产项目。核心链路已经闭环：飞书群消息采集、ByteHouse 入库、dashboard 生成、本地 Web 工作台、动态 API、人员身份补全、AI/人工复核输入、问数助手都已经接上；真实看板和 AI 结论边界也做了明确隔离。

需要诚实交接的问题也很明显：核心后端集中在一个 5,462 行文件里，前端集中在一个 4,790 行 HTML 文件里；dashboard 重建 19,188 条消息耗时约 232 秒；AI 分析结果是旧快照；当前目录不是 git 仓库；`.env`、`data/`、`people.json` 都包含敏感或隐私数据，交付前必须处理权限和密钥。

## 已完成能力

### 1. 飞书群消息采集

入口在 `src/group-analysis.js`：

- `sync`：增量同步飞书群消息。
- `backfill`：按天数或时间窗口回补历史消息。
- `groups-discover`：发现机器人所在群并写入 `groups.json`。
- `users-resolve`：通过飞书通讯录接口补全人员姓名、部门、身份。

已支持的消息字段：

- `messageId`
- `chatId`
- `projectId`
- `createTime`
- `sender`
- `msgType`
- `text`
- `mentions`
- `parentId` / `rootId`
- `replyToMessageId` / `threadRootMessageId`
- `raw` / `content`

当前不是只保存摘要，原始飞书返回内容也会保留，后续可重新做口径和规则。

### 2. 数据落地和存储

当前是多层存储：

- `data/messages.jsonl`：本地完整消息归档。
- `data/daily/YYYY-MM-DD/<chat_id>.jsonl`：按天和群拆分的归档。
- `data/reports/YYYY-MM-DD/summary.json`：日报聚合。
- `data/group-analysis.sqlite`：本地 SQLite 备份/兼容路径。
- ByteHouse：当前主查询和主看板存储。
- `data/dashboard/*.json`：前端工作台读的快照。

ByteHouse 已接入：

- messages 表
- sync runs 表
- sync chat runs 表
- people 表

`npm run db:status` 当前返回 ByteHouse 正常、消息数 19,188、people 表 68 人。

## 当前最真实的问题清单

### P0：交接前最好处理

1. 当前目录不是 git 仓库。

   `git status` 返回 `fatal: not a git repository`。这意味着没有清晰提交历史、版本 tag、回滚点。交给别人前至少应初始化仓库或放到正式代码仓库，并确认 `.env`、`data/`、`node_modules/` 不进入仓库。

2. `.env` 和真实数据在项目根目录。

   虽然 `.gitignore` 写了 `.env`、`people.json`、`data/`、`node_modules/`，但当前没有 git 仓库，实际交接时很容易把敏感数据整体打包出去。

3. AI 结果过期。

   `ai-insights.json` 是 2026-05-11 的 `openai/gpt-4o-mini` 结果，而当前数据已经到 2026-05-27，当前配置模型是 `deepseek/deepseek-chat-v3.1`。演示 AI 分析前应重新运行：

   ```bash
   npm run ai:analyze -- --run --write
   npm run dashboard
   ```

   如果不想花 token，就应该在交接说明里明确“AI 分析为旧快照，仅真实看板可作为当前事实”。

4. dashboard 重建性能较差。

   本次 `npm run verify` 中，`readAllMessages` 读取 19,188 条消息耗时约 232 秒。数据量再上去会更慢。对外演示时建议提前重建 dashboard，不要现场跑。

5. 4198 服务没有鉴权。

   当前服务只绑定 `127.0.0.1`，这对本机使用够了；但如果要部署到共享机器或公网，必须加鉴权、访问控制和日志脱敏。

### P1：接手后应该尽快优化

1. 后端大文件拆分。

   `src/group-analysis.js` 同时承担 CLI、飞书 API、ByteHouse、SQLite、dashboard、HTTP API、cron、AI 分析。建议拆成：

   - `src/config/`
   - `src/feishu/`
   - `src/storage/bytehouse.js`
   - `src/storage/sqlite.js`
   - `src/messages/normalizer.js`
   - `src/dashboard/build.js`
   - `src/server/routes/*.js`
   - `src/ai/analyze.js`
   - `src/cron.js`

2. 前端从单 HTML 拆出来。

   `public/index.html` 里同时有页面、样式、状态管理、渲染和业务规则，后续改 UI 很容易误伤。建议至少拆成：

   - `public/index.html`
   - `public/styles.css`
   - `public/app.js`
   - `public/api.js`
   - `public/render/*.js`

   如果后续继续演进，可以换成 Vite/React，但不是必须第一步。

3. ByteHouse 查询需要分页和聚合优化。

   当前 dashboard 重建会把全量消息读到 Node 内存再算，导致慢。应该优先把以下指标推到 SQL 聚合：

   - byDate
   - byHour
   - byProject
   - byChat
   - messageTypeRows
   - replyCount
   - uniqueSenderCount

   消息明细只在需要“完整消息列表”时分页读取。

4. ByteHouse 配置和查询逻辑重复。

   `src/group-analysis.js` 和 `src/agent/tools.js` 都有 ByteHouse 配置、SQL quote、查询函数。后续容易一边修了另一边没修。应抽成共享 storage adapter。

5. 错误处理有不少静默 fallback。

   代码里存在多处 `catch {}` 或失败后回退文件的逻辑。这保证了工具不容易挂，但也可能掩盖真实 ByteHouse 失败、JSON 坏行、前端加载失败。建议改成结构化 warning，并在 `/api/storage/status` 或页面顶部显式展示。

6. 问数助手路径不够干净。

   当前主路径是直连模型 API，Claude Agent SDK 工具调用路径保留但不可达。建议二选一：

   - 保留直连模型：删掉不可达 SDK 代码，把上下文检索逻辑做扎实。
   - 恢复 Agent SDK：让模型真的调用 `read_messages`、`search_messages`、`read_message_context`，并做工具权限控制。

7. `.env.example` 不完整。

   当前示例主要覆盖 Feishu、ByteHouse、AI analyze，但问数助手实际还用：

   - `AGENT_MODEL`
   - `OPENROUTER_API_KEY`
   - `ANTHROPIC_BASE_URL`
   - `ANTHROPIC_AUTH_TOKEN`
   - `AGENT_MAX_TOKENS`
   - `AGENT_SCRIPT_TIMEOUT_MS`
   - `REFRESH_STAGE_TIMEOUT_MS`

   建议补齐注释，否则接手人会配不起来问数助手。

8. 验收脚本不是完整测试体系。

   当前 `verify-requirements.js` 和 `verify-data-integrity.js` 很有价值，但它们是端到端验收，不是单元测试。建议补：

   - normalizer 单测
   - issue candidate 单测
   - ByteHouse SQL builder 单测
   - API route 单测
   - 前端关键渲染 smoke test

### P2：长期优化

1. 数据治理。

   需要明确哪些数据是原始事实、哪些是规则候选、哪些是 AI/人工结论，并给每种结论加 `source`、`generatedAt`、`model`、`evidenceIds`、`reviewer`。

2. 人员身份治理。

   目前 `people.json` 已有 68 人，其中 63 人来自飞书通讯录，3 人手工输入，2 人来自配置。后续需要处理外部人员、机器人、离职人员、部门变更、同名人员。

3. 项目事件能力还没真正用起来。

   `project-events.json` 当前基本为空。看板已经有项目节奏字段和 API，但运营要真实使用，还需要维护事件录入规范。

4. AI 结果需要版本化。

   当前 `ai-insights.json` 只有一份，会被保留或覆盖。建议按时间、模型、分析窗口保存历史：

   ```text
   data/dashboard/ai-runs/YYYY-MM-DDTHH-mm-ss/
     input.json
     output.json
     metadata.json
   ```

5. 部署方式缺失。

   当前是本机 Node 服务，没有 Dockerfile、systemd/launchd 配置、健康检查、日志轮转、备份策略。若要交给其他人长期维护，应补最小部署文档。

6. 安全和权限。

   如果部署给多人使用，需要：

   - 登录鉴权
   - 项目级权限
   - 敏感消息脱敏
   - API 访问日志
   - 模型调用审计
   - 防止任意人触发同步/AI 消耗

## 推荐交接前整理动作

建议按这个顺序做，能显著降低“别人发现代码很乱”的风险。

1. 初始化正式仓库，只提交代码、示例配置和文档。

   ```bash
   git init
   git add README.md package.json package-lock.json .env.example .gitignore src public people.example.json project-events.example.json 项目交接文档.md
   ```

   注意：当前没有 `groups.example.json`，需要先从 `groups.json` 脱敏生成，再单独加入仓库。

2. 不提交这些文件：

   ```text
   .env
   people.json
   project-events.json
   data/
   node_modules/
   output/
   .playwright-cli/
   ```

3. 补一份 `groups.example.json`。

   把真实 `chatId`、项目名替换成示例，保留结构即可。

4. 更新 `.env.example`。

   把问数助手相关变量补进去，注明哪些会产生模型费用。

5. 重新跑一次 AI 或清空旧 AI 结果。

   如果要展示 AI 页，重新跑：

   ```bash
   npm run ai:analyze -- --run --write
   npm run dashboard
   ```

   如果只展示真实看板，建议把 AI 页标成“旧结果，仅参考”或清空 `ai-insights.json`。

6. 写一个最小启动说明。

   README 现在比较完整，但接手人最需要的是：

   ```bash
   npm install
   cp .env.example .env
   npm run db:status
   npm run serve -- --port 4198
   open http://127.0.0.1:4198
   npm run verify
   ```

7. 给 dashboard 重建加醒目提示。

   现在 19,188 条消息重建用了约 232 秒，接手人容易以为卡死。建议在 README 和页面刷新按钮旁边写“全量重建约 4 分钟”。

## 接手人第一天 Checklist

1. 确认 Node 版本。

   当前本机使用 Node `v22.11.0`。建议 Node 20+。

2. 安装依赖。

   ```bash
   npm install
   ```

3. 配置 `.env`。

   需要至少配置飞书、ByteHouse、模型 API。没有模型 API 时真实看板可用，AI/问数能力不可用或降级。

4. 验证存储。

   ```bash
   npm run db:status
   ```

5. 启动服务。

   ```bash
   npm run serve -- --port 4198
   ```

6. 浏览器打开。

   ```text
   http://127.0.0.1:4198
   ```

7. 跑验收。

   ```bash
   npm run verify
   ```

8. 随机抽查三件事：

   - dashboard 总消息数是否等于 ByteHouse `messageCount`。
   - 某个项目的日历计数是否等于 `/api/metrics`。
   - 某个飞书通讯录人员在消息流里的姓名/部门是否和 `people.json` 一致。
